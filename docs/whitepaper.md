# Does GPU Cluster Topology Change What a GPU-Hour Is Worth?

An empirical study of NVLink domain size and MLPerf Training throughput

## Abstract

Cloud GPU pricing is quoted per GPU-hour, as if a GPU-hour were a fungible unit regardless of how the GPU is wired to its neighbors. This paper tests that assumption using six rounds of public MLPerf Training results (v3.1–v6.0, n=969 benchmark submissions after cleaning, 41 submitting organizations). We regress log time-to-train on log total GPU count, log NVLink domain size, model fixed effects, and generation fixed effects, with organization-clustered standard errors. Holding total GPU count fixed, doubling the NVLink domain size is associated with 6.5–13.1% less time-to-train, depending on specification (categorical domain-size coefficient -0.0961, p=0.026; mined-evidence-augmented coefficient -0.1309, p=0.0001), a result that survives wild cluster bootstrap inference, Huber and median-regression estimators, a temporal holdout, and a leave-one-generation-out sensitivity sweep. Splitting the effect by measured communication intensity shows the discount concentrates in communication-bound workloads (comm-bound: domain-doubling multiplier 0.894 on time-to-train; comm-light: 0.943). We then compare this measured performance premium to a market-price premium built from 13 real GPU rental SKUs across AWS, Azure, and OCI: bigger NVLink domains cost 1.6–1.8x more per GPU-hour, a spread that exceeds the 0.89x-1.00x performance discount even after adjusting for GPU generation (net combined cost-to-train multiplier of 1.11–1.58x, i.e., renting the larger domain is not paid back by its speed advantage on this snapshot). We report every estimate alongside its identification caveats, the most important of which is that public MLPerf submissions are not a random sample of possible topology-workload pairs.

## Introduction

Compute markets price a GPU-hour as a single fungible unit: $X per GPU per hour, the same $X whether that GPU sits alone or as one of 72 GPUs sharing a single NVLink-connected domain. But large-scale training jobs are not embarrassingly parallel — they synchronize gradients, exchange activations across pipeline stages, and shard optimizer state across devices, all of which move data over the interconnect fabric. If a wider NVLink domain lets a training job finish the same amount of work in less wall-clock time, then a GPU-hour on a 72-GPU domain and a GPU-hour on a standalone GPU are not economically equivalent units — they are different goods being sold under the same name.

This matters concretely for anyone buying training compute. If topology genuinely changes throughput per dollar, then GPU-hour price comparisons across providers or instance types are comparing incommensurate quantities, the same failure mode as comparing gasoline prices without ever converting to price-per-mile. It also matters for hardware buyers and cluster architects deciding whether to pay a premium for tightly-coupled fabric versus a larger number of loosely-coupled nodes, and for anyone trying to build a fair total-cost-of-ownership model for a training run rather than a total-cost-of-renting model.

The empirical question is straightforward to state and hard to answer cleanly with only public data: does NVLink domain size — the number of GPUs that can talk to each other over NVLink/NVSwitch without going through a slower network fabric — causally reduce time-to-train, independent of total GPU count and workload? MLPerf Training is the only public dataset with enough submissions, spanning enough hardware generations and topologies, to attempt an answer without running a single benchmark ourselves. Every submitted result already reports total accelerator count, system configuration, and wall-clock time-to-converge for a fixed target quality, checked against a fixed reference model. That gives us the two things we need — a time-to-train outcome and a topology covariate — for 969 submissions after cleaning.

This paper documents what that regression can and cannot show. We report a topology premium that is small in absolute magnitude, statistically robust across multiple inference procedures and functional-form choices, concentrated in communication-bound workloads as the underlying mechanism would predict, and — when compared against real market pricing — currently outweighed by the price premium providers charge for wider domains. We are equally explicit about the three respects in which this remains an observational study of a non-random, provider-curated sample, not a controlled experiment, because those limitations bound how the result should be used.

## Data

The dataset is built entirely from `systems/*.json` submission metadata and result-log timestamps published by MLCommons in the official MLPerf Training results repositories, six rounds from v3.1 through v6.0 (`scripts/01_extract_mlperf_data.py`). Each row is one system-model benchmark result: an organization's system, running one reference model, achieving one time-to-train. For every row we record total accelerator count (`total_gpus`), accelerators per node, an NVLink domain-size field, the model being trained, the accelerator generation, and the submitting organization (used for clustering).

Domain size is not a single clean field across the whole corpus, so it is constructed three ways, in increasing order of trustworthiness. **v1** uses accelerators-per-node as a domain-size proxy — cheap to compute but wrong wherever NVLink spans multiple physical nodes (as GB200/GB300 NVL72 racks do). **v2** ("corrected") replaces the proxy with a categorical NVLink domain field, applying a flat `min(total_gpus, 72)` cap specifically to GB200/GB300 rows, since NVL72 racks top out at 72 GPUs of direct NVLink connectivity regardless of how many nodes are aggregated. That flat cap is itself an approximation for the 92 dataset rows (47 unique submissions) where total GPU count is below 72 — a "sub-rack" deployment might have less than the full 72-GPU domain, or the cap might still be right. To resolve this without guessing, `scripts/07_mine_nvl_configs.py` mines each submission's own `system_name` string and benchmark config filenames for direct topology evidence, distinguishing genuine rack-fabric tokens (e.g., "GB200 NVL72" in a system name) from tray/node-shape tokens that look similar but describe something else (e.g., "NVL4" in some GB300 system names turns out to mean 4 accelerators per tray, not a 4-GPU NVLink fabric — confirmed by cross-referencing against `accelerators_per_node`). Of the 92 sub-rack rows, this mining directly confirms the flat cap for 54 (31 unique submissions) with no conflicting evidence found anywhere in the corpus, and leaves 38 (16 unique submissions) as cap-only, unresolved by direct evidence (`results/nvl_mining_reconciliation.txt`). The **mined-augmented** domain field overrides the categorical cap with mined evidence wherever confidence is high (137 rows total, including higher-domain-count generations beyond the 92 GB200/GB300 sub-rack set).

A separate, script-based measurement replaces the original hand-labeled `COMM_BOUND` binary workload tag with `log(TP×PP)`, the product of tensor- and pipeline-parallelism degree, mined directly from benchmark configuration filenames rather than a subjective label (`scripts/11_measured_comm_intensity.py`). This is only recoverable for models that expose TP/PP in their config naming (7 of the corpus's models); coverage across all 969 rows is 31.3% measured, 24.8% assumed data-parallel by convention, 6.7% unmeasured due to scale mismatch, and 37.3% unavailable because the relevant benchmark implementation directory isn't present in the cloned rounds. The restricted measured-or-assumed-DP subsample used for comm-intensity analysis is n=543.

## Methods

The headline specification regresses log time-to-train on log total GPU count, log NVLink domain size, a full set of model fixed effects, and a full set of accelerator-generation fixed effects, with standard errors clustered by submitting organization (41 clusters):

`log_time ~ log_total_gpus + log_domain_size + model_FE + gen_FE`

Including `log_total_gpus` alongside `log_domain_size` is the load-bearing modeling choice: it reparametrizes the regressor space so the domain-size coefficient measures the topology effect holding total scale fixed, rather than conflating domain size with how many total GPUs are being used (a naive `log_domain_size + log_domain_count` basis spans the same two-dimensional space algebraically, but its domain-size coefficient lets total GPU count vary with domain size, which is why that basis produces a much larger, harder-to-interpret coefficient — see Results). Model and generation fixed effects absorb the fact that different models have very different absolute training times and different hardware generations have different raw throughput, so the domain coefficient is identified from variation in topology conditional on model and generation, not from the fact that newer generations happen to ship in bigger domains.

Because the effective sample size for clustered inference is the number of clusters (41 organizations, not 969 rows), asymptotic cluster-robust standard errors can be unreliable in exactly the regime this dataset sits in. We additionally compute wild cluster bootstrap-t p-values (Rademacher weights, null imposed, 9999 replications, via the `wildboottest` package) for the two headline specifications, to check whether the asymptotic p-values are overstating precision (`scripts/10_inference_hardening.py`).

To test whether the topology effect is a workload-dependent communication-latency story rather than a generic artifact, we interact domain size with a communication-intensity term, first the hand-labeled binary and then the measured `log(TP×PP)` continuous version, and re-estimate on the restricted subsample where the continuous measure is available (`scripts/11_measured_comm_intensity.py`). The final discount function used for the calculator fits two separate slopes — one for comm-light, one for comm-bound workloads — against the mined-augmented domain field, and reports the implied discount at standard domain sizes (4/8/16/36/72) relative to a domain=4 baseline, with monotonicity enforcement (clipping if a fitted curve ever implied a wider domain being slower) and organization-clustered 95% bands (`scripts/09_discount_function.py`).

For the market comparison, list prices for 13 GPU rental SKUs were collected live from AWS, Azure, and OCI pricing APIs (`scripts/12_market_price_collection.py`; vast.ai and GPUPerHour were excluded per their terms of service), and regressed log-log against domain size, both pooled and with provider fixed effects. A second version of this comparison converts raw $/GPU-hour to $/effective-GPU-hour using generation speed multipliers estimated from the mined-augmented regression's own generation fixed effects, then propagates the sampling uncertainty in those four generation coefficients through a 5,000-draw Monte Carlo simulation (multivariate normal draws from the estimated coefficients and their organization-clustered covariance matrix, seed fixed at 42) rather than a delta-method approximation, since the price ratio is a nonlinear function of the coefficients (`scripts/14_gen_adjusted_hedonic.py`).

## Results

**Premium estimates across specifications.** The naive v1 specification (accelerators-per-node as the domain proxy) gives a domain-size coefficient of -0.3316 (nonrobust SE 0.0497, p=4.5e-11; organization-clustered SE 0.0900, p=2.3e-04). Once domain size is corrected to the categorical NVLink field with the GB200/GB300 sub-rack cap, the coefficient shrinks by roughly 3.5x, to -0.0961 (nonrobust SE 0.0237, p=5.4e-05; clustered SE 0.0432, p=0.026) — v1 was overstating the effect by conflating "more GPUs per node" with genuine topology, since node-count and domain-size proxies diverge exactly for the multi-node NVL72 systems that dominate the recent, largest-scale submissions (n=969, 41 clusters, R²=0.9562, both specs). Substituting the mined-evidence domain field for the categorical cap tightens the estimate further, to -0.1309 (clustered SE 0.0339, p=0.0001; R²=0.9565), with a leave-one-generation-out sensitivity sweep across all 14 generations in the corpus ranging from -0.1028 (dropping H100) to -0.1861 (dropping GB300) — the sign and rough magnitude hold regardless of which single generation is excluded. Wild cluster bootstrap-t inference confirms both headline estimates remain significant at α=0.05 (categorical: asymptotic p=0.026 → bootstrap p=0.028; mined-augmented: asymptotic p=0.0001 → bootstrap p=0.0067), though the bootstrap p-value is visibly larger for the mined-augmented spec, meaning the 41-cluster asymptotic SEs were somewhat optimistic there. (For reference, a non-fixed-scale basis — regressing on log domain size and log domain count rather than log domain size and log total GPUs — gives much larger coefficients, -1.0221 for v1 and -0.7806 for v2, because that basis lets total GPU count move with domain size; those numbers describe a different, scale-conflated quantity and are reported here only for AIC/BIC model-comparison context, not as premium estimates.)

The estimate is stable under alternative estimators (Huber M-estimator: -0.0837, SE 0.0167; median/quantile regression: -0.0898, SE 0.0150) and under a temporal holdout that fits on early rounds (v3.1–v5.0, n=231) and evaluates on later ones (v5.1–v6.0, n=143): early-fit coefficient -0.1208 (p=0.057), late-fit coefficient -0.3101 (p=0.021), with the early-fit coefficients achieving out-of-sample R²=0.886 and median absolute error 17.2% when used to predict the late rounds. Restricting to a "clean identification" subsample (n=506) gives -0.3028 (SE 0.0877, p=0.0006) — larger than the full-sample estimate, consistent with noise being added, not bias being removed, when the fuller sample is used. Using median training-round time instead of a single logged run (excluding PCIe-connected systems entirely) gives -0.0866 (p=0.062, n=969) and -0.0826 (p=0.038, n=934) respectively, both directionally consistent with the headline.

**Mechanism: communication intensity.** If the topology effect reflects communication cost rather than a generic scale artifact, it should concentrate in communication-bound workloads. The measured continuous comm-intensity spec, fit on the n=543 subsample where TP×PP is directly recoverable, finds a domain main effect of -0.1011 (p=0.0006) and a domain×comm-intensity interaction of -0.0384 (p=0.004) — the interaction is more significant than the equivalent interaction under the original hand-labeled binary (p=0.023 on the same subsample), i.e., the measured proxy sharpens rather than weakens the mechanism story. The final discount function (fit on the full mined-augmented domain field) gives separate comm-light and comm-bound slopes of -0.0841 (p=0.058) and -0.1618, respectively, translating into the following time-to-train multipliers relative to domain=4 (organization-clustered 95% bands in brackets):

| Domain | Comm-light multiplier | Comm-bound multiplier |
|---|---|---|
| 4 | 1.000 [1.000, 1.000] | 1.000 [1.000, 1.000] |
| 8 | 0.943 [0.888, 1.002] | 0.894 [0.850, 0.940] |
| 16 | 0.890 [0.789, 1.004] | 0.799 [0.723, 0.884] |
| 36 | 0.831 [0.687, 1.007] | 0.701 [0.598, 0.822] |
| 72 | 0.784 [0.610, 1.009] | 0.627 [0.508, 0.773] |

A temporal holdout on this discount function (same early/late split as above) gives out-of-sample R²=0.892 and median error 17.4%. A robustness check that reclassifies the one model (flux1) whose hand-label conflicts with its measured zero TP/PP parallelism moves individual table cells by at most 0.017, far inside the width of the reported 95% bands (up to 0.40 wide), so this discount table is not sensitive to that labeling choice.

**Market comparison.** 13 GPU rental SKUs across AWS, Azure, and OCI, spanning domain sizes 1 (n=6, median $0.742/GPU-hr), 8 (n=4, median $8.440/GPU-hr), and 72 (n=3, median $18.000/GPU-hr), give a pooled price-domain slope implying a 1.764x price increase per domain doubling (R²=0.797; 1.771x with provider fixed effects, R²=0.858). Paired against the comm-bound throughput multiplier of 0.894x per domain doubling, the combined cost-to-train multiplier is 1.577x (pooled) to 1.583x (provider-FE): on this snapshot, wider domains cost more overall even after finishing faster, because the market's price spread exceeds the measured performance spread. Because domain size and GPU generation are confounded in this raw comparison (no provider sells the same GPU model at more than one domain size), we repeat it after excluding SKUs with no measured MLPerf generation (T4, L4, A10/A10G — 5 of 13 rows) and converting the remaining 8 rows' prices to $/effective-GPU-hour using generation speed multipliers from the mined-augmented model's own generation fixed effects (H100: 3.260x A100; B200: 6.886x; GB200: 5.922x; GB300: 8.446x). On this generation-adjusted, n=8 subset, the raw price premium is 1.619x (pooled) and the effective-price premium after generation adjustment falls to 1.243x [95% simulated interval: 1.180, 1.306] (1.234x with provider FE [1.172, 1.297]). Combined with the same 0.894x throughput multiplier, this yields cost-to-train multipliers of 1.448x (raw price, n=8 subset) and 1.111x (generation-adjusted price) — the direction is unchanged from the full n=13 comparison, but the magnitude shrinks substantially once the generation confound is removed from the price side.

## Limitations

Three limitations are inherent to what can be learned from public MLPerf submissions, not artifacts fixable by more careful regression:

1. **Topology assignment is not random.** Organizations choose what topology to submit alongside what model, hardware generation, and scale, presumably in ways correlated with unobserved factors (available fabric, internal priorities, marketing goals). No fixed-effects specification can fully rule out that some part of the domain-size coefficient reflects these correlated choices rather than a pure causal topology effect. The temporal holdout, alternative estimators, and clean-identification subsample analyses in Results are attempts to bound this concern, not to eliminate it.

2. **Identification is thin and concentrated.** Only 37.3% of the total variance in the regression is within-generation variance (as opposed to explained by generation fixed effects), and a large share of the domain-size identification for the two most recent generations comes specifically from the 92 GB200/GB300 rows with total GPU count below 72 (the "sub-rack" deployments), out of 187 GB200/GB300 rows overall. The mined-evidence check (`results/nvl_mining_reconciliation.txt`) confirms the categorical domain cap directly for 54 of those 92 rows with no conflicting evidence anywhere in the corpus, but 38 remain cap-only, unresolved by direct textual evidence. A sensitivity check that drops all 38 cap-only rows produces a substantially larger coefficient (-0.3376, SE 0.0910, n=931) — but a Cook's-distance leverage audit finds that only 1 of the 10 highest-leverage rows in that reduced sample is actually a mined-evidence row, meaning the larger coefficient is driven mostly by which rows happen to remain, not by the quality of the evidence. That number should not be read as a more-confirmed estimate.

3. **Submission portfolios are self-selected.** Organizations submit the results that make their hardware and configuration look best; the corpus is not a random sample of all topology-workload pairs that were ever run, only the ones organizations chose to publish. This selection is not directly observable or correctable from public data alone.

A separate, additive limitation applies specifically to the market comparison: the raw n=13 SKU snapshot shrinks to n=8 once GPU models absent from the measured MLPerf generations (T4, L4, A10/A10G) are excluded rather than mapped to an approximate substitute, and even among the 8 remaining rows, domain size and GPU generation remain perfectly confounded (domain=1 is only A100, with n shrinking to a single row; domain=8 is only H100/B200; domain=72 is only GB200/GB300). No within-generation domain variation exists anywhere in the market snapshot, so the market-side slope is a descriptive first-order comparison, not a matched causal estimate, regardless of the generation adjustment applied to the price side. Finally, the collected prices are public list prices, not verified transaction prices, and reflect a single point-in-time snapshot rather than a time series.

## Reproducibility

The full analysis, including all scripts referenced above, is public at `github.com/elyryba/gpu-topology-efficiency`. The repository's `Reproduce` GitHub Actions workflow re-runs the core extraction and modeling pipeline (scripts 02 through 06) against pinned dependencies (`requirements.txt`, Python 3.9) on every push and on a monthly schedule, then runs a pytest suite (`tests/test_reproduction.py`) that locks the headline coefficients, standard errors, and sample sizes reported here against silent numerical drift. Scripts that require the raw MLPerf submission repositories (07, mining) or that hit live pricing APIs (12) are excluded from the automated CI run, since they either need a local clone of the source repositories or represent a point-in-time price snapshot that shouldn't silently re-collect on every push, but are runnable manually and are documented in the repository's `CLAUDE.md`. A live discount calculator built on the mined-augmented discount function is deployed via GitHub Pages at `elyryba.github.io/gpu-topology-efficiency`, sourced from a versioned `model_params.json` exported by script 15 and tagged to the same release as this paper.
